# 英语单词微信小程序接口文档

## 1. 基本信息

- 项目名称：英语单词微信小程序
- 接口版本：`/api/v1`
- 后端框架：Django + Django REST Framework
- 数据库：MySQL 8
- 认证方式：Token

## 2. 请求约定

### 2.1 Base URL

本地开发默认地址：

```text
http://127.0.0.1:8000/api/v1
```

### 2.2 认证头

登录成功后，请在请求头中携带：

```text
Authorization: Token {token}
```

### 2.3 统一响应格式

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

### 3.2.1 微信登录

- 方法：`POST`
- 路径：`/api/v1/auth/wx-login`
- 认证：否

请求示例：

```json
{
  "code": "demo_user",
  "nickname": "Tester"
}
```

说明：

- 当前版本为开发环境模拟登录，`code` 会被转换成 `mock_{code}` 作为用户唯一标识。
- 小程序登录页在开发调试模式下支持填写“测试账号标识”。只要多次使用同一个标识，例如 `local_tester`，前端就会提交固定的 `debug_local_tester`，后端仍会命中同一个 `mock_debug_local_tester` 用户，因此即使清空模拟器缓存，学习记录、计划、错词本和个性化 RAG 状态也会保留在同一测试账号下。

## 3.4 学习计划接口

### 3.4.1 获取当前计划

- 方法：`GET`
- 路径：`/api/v1/plans/current`
- 认证：是
- 说明：返回最近一个可管理计划，优先返回 `active`，没有 active 时返回最近的 `paused` 计划，便于小程序恢复暂停计划。

### 3.4.2 创建学习计划

- 方法：`POST`
- 路径：`/api/v1/plans`
- 认证：是

请求示例：

```json
{
  "book_id": 1,
  "daily_target": 20
}
```

### 3.4.3 更新当前计划

- 方法：`PUT`
- 路径：`/api/v1/plans/current`
- 认证：是

请求字段：

- `daily_target`：每日学习目标，支持 `1` 到 `200`
- `status`：计划状态，支持 `active`、`paused`、`completed`

说明：

- 当 `daily_target` 更新时，后端同步更新用户设置中的 `daily_target`。
- 当计划从 `paused` 恢复为 `active` 时，后端会暂停同用户其他 active 计划并重新生成今日任务。

### 3.4.4 切换词书

- 方法：`POST`
- 路径：`/api/v1/plans/current/switch-book`
- 认证：是

请求示例：

```json
{
  "book_id": 2,
  "daily_target": 30,
  "keep_progress": true
}
```

说明：

- `keep_progress=true` 时会把旧计划已完成数量带入新计划。
- `keep_progress=false` 时新计划从当前词书重新开始。

## 3.6 复习接口

### 3.6.1 获取复习任务

- 方法：`GET`
- 路径：`/api/v1/review/tasks`
- 认证：是

支持查询参数：

- `limit`

题型说明：

- `meaning_to_word`：释义选词
- `word_to_meaning`：单词选释义
- `example_to_word`：例句选词
- `spelling`：拼写填空
- `listening_to_word`：听音辨词，返回 `speech_text`、`speech_lang`，小程序端调用 `/system/speech` 时传入全局用户设置中的 `speech_speed`

### 3.6.2 提交复习结果

- 方法：`POST`
- 路径：`/api/v1/review/submit`
- 认证：是

请求示例：

```json
{
  "session_id": 1,
  "answers": [
    {
      "word_id": 1,
      "user_answer": "resilient",
      "question_type": "meaning_to_word"
    }
  ]
}
```

响应说明：

- `answers[].answer_feedback`：提交后立即返回的答题讲解，包含 `status`、`title`、`explanation`、`recovery_tip`、`similarity`、`quality` 等字段。
- 小程序复习页应先展示本题反馈，再进入下一题；错题反馈会同时给出正确答案和恢复建议。

### 3.6.3 获取复习结果

- 方法：`GET`
- 路径：`/api/v1/review/result/{session_id}`
- 认证：是

说明：

- `records[].answer_feedback` 会返回每题提交时保存的即时反馈，便于后续结果页复盘。

`/api/v1/ai/evaluations` 返回补充字段：

- `summary.run_count`：本次汇总的评测运行数
- `summary.pass_rate`：通过率
- `summary.regression_report`：每个 case 的最近一次与上一次结果对比
- `summary.prompt_version_compare`：每个 case 目前出现过的 prompt version
- `summary.failed_replays`：最近失败样本，可作为回放入口

`/api/v1/ai/rag/vector-search` 返回补充字段：

- `documents[].rerank_score`：rerank 后得分
- `documents[].rank_debug`：得分构成调试信息
- `retrieval_explain.rerank_enabled`：当前是否启用 rerank
- `retrieval_explain.multi_route_enabled`：当前是否启用多路召回

`/api/v1/ai/rag/recall-evaluate` 返回补充字段：

- `hybrid_recall`：hybrid 检索覆盖率
- `retrieval_runtime.vector`：向量检索运行时
- `retrieval_runtime.hybrid`：hybrid 检索运行时

`/api/v1/ai/scenario/dialogue` 请求示例：

```json
{
  "scenario": "restaurant",
  "user_message": "I want order a coffee.",
  "conversation_id": 1
}
```

统一返回补充字段：

- `headline`
- `summary`
- `feature_runtime`
- `runtime_stack`
- `context_sources`
- `langchain_trace`
- `evidence`

说明：

- 轻量 AI 场景和主 Agent 场景都已尽量向这套字段收敛，前端可以按统一协议展示结果摘要、运行路径和证据区。
- `feature_runtime.path` 用于区分当前请求主要走的是 `langgraph`、`langchain_explicit`、`direct`、`fallback` 等哪条链路。
- `context_sources` 用于告诉用户“本次建议来自哪些上下文”，例如当前计划、错词本、学习画像、个性化 RAG 等。

## 6. 2026-05 AI 接口补充说明

### 6.1 学习计划重规划 Agent

- 方法：`POST`
- 路径：`/api/v1/ai/plans/replan`
- 认证：是

请求示例：

```json
{
  "trend_days": 7
}
```

返回要点：

- `headline`：本次重规划标题
- `summary`：本次调整摘要
- `new_plan`：新的日目标、复习目标、模式与建议顺序
- `plan_patch`：可直接应用的计划补丁
- `decision`：关键决策原因
- `agent_flow`：Agent 工作流步骤
- `knowledge`：结构化 RAG 与向量 RAG 的知识结果
- `evidence`：统一证据区数据
- `multi_agent`：角色、handoff、已选工具或已选路径原因
- `langchain_trace`：显式 LangChain / parser / tool trace
- `runtime_stack`：本次运行栈摘要

### 6.2 统一 AI evidence 返回结构

多数 AI 接口现在会附带 `evidence` 字段，常见结构包括：

- `summary`
- `workflow`
- `tools_used`
- `retrieval_hits`
- `observability`

说明：

- `workflow` 用于展示 AI / Agent 的执行步骤
- `tools_used` 用于展示 MCP tool 或内部工具调用
- `retrieval_hits` 用于展示结构化 RAG / 向量 RAG 命中的知识片段
- `observability` 用于展示模型、prompt、cache、latency、endpoint 等运行信息

### 6.4 AI 观测与评测补充

`GET /api/v1/ai/observability` 返回补充字段：

- `runtime_path_summary`：按 `langgraph / langchain_explicit / mcp_tool / chroma / personalized_rag / fallback` 聚合最近运行路径
- `feature_summary`：按功能统计请求量与缓存命中
- `prompt_versions`：近期 prompt version 使用情况
- `model_summary`：近期模型使用情况
- `recent_logs`：最近请求日志

`GET /api/v1/ai/evaluations/runs/{run_id}` 返回补充字段：

- `request_payload`
- `result_payload`
- `trace_payload`
- `runtime_snapshot`
- `feature_runtime`
- `runtime_stack`
- `replay_payload`

`POST /api/v1/ai/evaluations/runs/{run_id}/replay` 说明：

- 用已有 run 的 `request_payload` 重新执行一次同类评测
- 可通过 `request_payload` 覆盖部分参数
- 会生成新的评测运行记录，便于前端直接展示“单条失败样本重放”
