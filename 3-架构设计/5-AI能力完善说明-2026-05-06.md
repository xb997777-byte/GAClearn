## 6. 当前项目里的 RAG 真实状态说明

### 6.0 2026-05-06 新补充：标准 RAG 版本已经落地到 Chroma

当前项目的向量 RAG 已经补上了一条更标准的工程链路：

1. 从项目教学数据表抽取知识内容
2. 按词条 / 例句 / 语法点 / 语法句子构建知识 chunk
3. 用 embedding 模型把 chunk 转成向量
4. 将向量写入本地持久化的 Chroma 集合
5. 用户提问时先把 query 向量化，再到 Chroma 做相似度检索
6. 把召回的知识块交给 AI 生成 grounded answer

对应实现位置：

- 知识切块：`backend/apps/ai/rag/knowledge_base.py`
- Chroma 运行时：`backend/apps/ai/rag/chroma_runtime.py`
- 索引重建命令：`backend/apps/ai/management/commands/rebuild_rag_index.py`
- 索引状态服务：`backend/apps/ai/rag/index_status.py`
- 索引状态接口：`GET /api/v1/ai/rag/index-status`

如果 Chroma 或 embedding 依赖没有安装成功，系统会自动回退到原来的本地轻量向量检索，不会把整个 AI 中心直接跑挂。

另外，当前实现还补了一个更务实的降级策略：

- 优先使用在线 embedding API
- 在线 embedding 不可用时，退回本地 `sentence-transformers`
- 如果本地首次下载模型受网络影响，则临时降级为离线 hash embedding
- 即使降级，仍然保留 Chroma 的知识切块、入库、索引和检索链路

这样项目演示时不会因为外网模型下载卡住而完全不可用。

### 6.0.1 标准 RAG 的知识库数据来源是什么

这件事现在已经明确了，不再是“后面再说”的状态。

第一版标准 RAG 的知识库，直接来自你项目自己的核心教学数据表：

- `words`：单词主表
- `word_examples`：单词例句表
- `grammar_points`：语法点表
- `grammar_sentences`：语法句子表

也就是说，这个项目的 RAG 不是去抓外部网页，也不是先塞一堆无关 PDF，而是优先把你这个英语学习产品自己最核心的教学内容，变成可检索知识库。

这样做有三个好处：

1. 和产品本身强相关，问出来的答案不会偏题
2. 用户在页面里能真实感受到“这是在用我的学习内容回答我”
3. 你写进简历时可以明确说出“RAG 直接服务于业务知识库”，而不是只有一个抽象 demo

### 6.0.2 为什么第一版先选这些表

因为这四类数据正好覆盖了英语学习里最容易被 AI 检索增强的内容层：

- 单词解释层
- 单词使用场景层
- 语法规则层
- 语法句子示例层

它们结构稳定、字段清晰，而且本来就在你的业务数据库里，特别适合做第一版标准 RAG。

后续可以继续扩充到：

- AI 写作范文与批改样例
- 翻译训练材料
- 场景对话模板
- 用户个人薄弱点总结
- 学习计划历史与阶段性错因知识库

但第一步不建议一上来塞太多来源，否则知识库会很杂，用户体验也容易变得不稳定。

### 6.2 你对 RAG 的理解对不对

你的理解是对的，而且已经非常接近标准生产级 RAG 流程：

1. 先准备一批资料
2. 把资料切块
3. 把每个 chunk 转成 embedding 向量
4. 存进向量数据库
5. 用户提问时把问题也转成向量
6. 去向量数据库里做相似度检索
7. 把召回结果连同问题一起发给大模型
8. 让模型基于召回结果生成 grounded answer

这就是更标准的向量 RAG。

### 6.3 当前项目里是怎么实现 RAG 的

当前项目里其实有两套 RAG：

#### 1）结构化 RAG

文件位置：

- `backend/apps/ai/learning_assistant.py`

核心实现：

- `retrieve_learning_context(...)`
- `run_rag_search(...)`

它的做法不是向量检索，而是：

1. 从用户问题里抽关键词
2. 去 `words`、`grammar_points`、`grammar_sentences` 这些表里做数据库检索
3. 把检索到的词条、语法点、句子整理成上下文
4. 交给模型生成回答

所以结构化 RAG 更像：

“先按关键词查词库和语法库，再根据查到的内容回答。”

#### 2）标准 Chroma 向量 RAG（当前主路径）

文件位置：

- `backend/apps/ai/rag/knowledge_base.py`
- `backend/apps/ai/rag/chroma_runtime.py`
- `backend/apps/ai/management/commands/rebuild_rag_index.py`
- `backend/apps/ai/learning_assistant.py`

核心实现：

- `build_all_knowledge_chunks(...)`
- `embed_texts(...)`
- `search_knowledge_base(...)`
- `rebuild_rag_index`
- `run_vector_rag_search(...)`

它现在的流程是：

1. 从 `words`、`word_examples`、`grammar_points`、`grammar_sentences` 提取教学内容
2. 按知识单元构建 chunk
3. 用 embedding 模型把 chunk 转成向量
4. 写入本地持久化 Chroma 集合
5. 用户 query 转成向量后到 Chroma 检索相近 chunk
6. 把召回的知识块交给模型生成 grounded answer

这就是你说的那套“切块 -> 向量化 -> 入库 -> 检索 -> 带上下文回答”的标准 RAG 路线。

#### 3）本地轻量向量 RAG（当前 fallback 路径）

文件位置：

- `backend/apps/ai/rag/vector_runtime.py`
- `backend/apps/ai/learning_assistant.py`

核心实现：

- `build_local_vector_documents()`
- `vectorize_text(...)`
- `cosine_score(...)`
- `LocalHashVectorStore.search(...)`
- `run_vector_rag_search(...)`

它现在的流程是：

1. 从词条、语法点、例句中动态拼出本地文档
2. 用 token 计数的方式做轻量“向量化”
3. 对用户 query 做同样的 token 向量化
4. 用余弦相似度排序
5. 取前几条资料交给模型生成回答

注意：

这一路现在不再是主路径，而是 Chroma 不可用时的保底方案。

### 6.7 新补充：个性化 RAG 改成用户显式授权 + 手动建库

这次把“用户个人记录参与 RAG”这件事改成了更清晰也更适合演示的方式：

1. 默认关闭，不会自动把用户学习记录做成个人知识库
2. 用户需要先在“学习设置”里主动开启个性化 RAG 授权
3. 开启后仍然不会自动建库，必须再点击“创建 / 刷新我的个性知识库”
4. 后端才会基于以下信息构建用户个人知识块：
   - 当前学习计划
   - 今日任务与计划调整记录
   - 错词本与最近错题
   - 最近学习行为
   - 最近语法薄弱点
5. 构建结果写入用户独立的 Chroma collection，而不是和公共教学知识库混在一起
6. 向量检索时会把“公共知识库 + 用户个人知识库”一起纳入召回，并在返回结果中标明个性化命中数量与来源

这样做的好处是：

- 用户视角更容易理解：这是一个“我允许 AI 使用我的学习记录来增强回答”的能力
- 产品边界更清楚：不是默认吃用户数据，而是显式授权
- 项目展示更强：你可以明确讲出“个性化 RAG 的授权、建库、召回、展示”完整链路
- 简历表述也更自然：不仅有公共业务知识库 RAG，还有用户级 personalized retrieval
