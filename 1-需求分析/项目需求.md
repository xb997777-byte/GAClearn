# 英语单词微信小程序项目需求文档

## 1. 文档说明

- 项目名称：英语单词微信小程序
- 文档类型：项目需求文档
- 当前版本：V1.2
- 编写日期：2026-04-28
- 适用范围：微信小程序端、后端 API、后续管理后台规划
- 技术栈约束：后端 API 使用 Python Django 框架，数据库使用 MySQL 8

## 2. 项目背景

面向有英语单词记忆需求的用户，提供一个基于微信小程序的轻量化学习工具，帮助用户完成选词书、制定学习计划、每日背词、复习巩固、测试评估和学习数据追踪。

相比传统单词 App，本项目优先强调以下特点：

- 使用门槛低：无需安装独立 App，直接通过微信使用。
- 学习节奏清晰：支持每日学习目标和复习任务。
- 内容闭环完整：学习、复习、测试、错词回看形成完整流程。
- 数据可跟踪：记录学习进度、掌握程度和连续打卡情况。

## 3. 项目目标

### 3.1 产品目标

- 为用户提供一个可持续使用的英语单词学习工具。
- 支持用户按词书和计划进行系统化学习。
- 支持通过真实句子、长难句拆解和颜色标注进行语法学习，帮助用户建立“单词 + 语法 + 句子理解”的一体化能力。
- 通过复习机制提升记忆效果，减少学完即忘的问题。
- 通过数据统计提升用户学习反馈感和持续性。
- 逐步引入 AI 助学能力，让系统不仅能展示内容，还能根据用户状态做讲解、陪练、复盘和教练式建议。

### 3.2 业务目标

- 完成微信小程序 MVP 版本上线。
- 支持基础用户增长、留存和学习行为分析。
- 为后续扩展会员、专项词库、AI 助学等能力预留空间。
- 建立 AI 助学二期规划，为后续接入 LangChain、LangGraph、RAG、MCP 等能力提供产品边界。

## 4. 用户角色

### 4.1 学习用户

主要使用微信小程序进行单词学习的个人用户，典型人群包括：

- 中学生
- 大学生
- 考研用户
- 英语考试备考用户
- 职场英语学习用户

### 4.2 运营/管理员

首版不强制要求独立管理后台上线，但后端需要预留词书、单词、例句、标签、学习数据查询等扩展能力，便于后续接入运营后台。

## 5. 项目范围

### 5.1 本期建设范围

- 微信小程序端
- 面向小程序端的后端 API
- 用户登录与身份体系
- 词书与学习计划
- 每日学习任务

- 语法学习与长难句拆解

### 5.2 暂不纳入首版范围

- 社区互动

- AI 生成例句/释义
- Web 管理后台完整页面
- 多语言国际化

### 5.3 技术选型约束

- 小程序端采用微信原生小程序方案。
- 后端 API 固定使用 Python + Django 技术栈开发。
- API 层建议使用 Django REST Framework 实现，以提升接口开发效率和可维护性。
- 数据库固定使用 MySQL 8.0。
- 数据访问层使用 Django ORM。
- 数据库字符集统一采用 `utf8mb4`，以保证中英文、特殊符号和扩展字符兼容性。

## 6. 核心业务流程

### 6.1 首次使用流程

1. 用户进入小程序。
2. 用户完成微信授权登录。
3. 系统引导用户选择词书。
4. 用户设置每日学习目标，例如每天 20 个单词。
5. 系统生成当日学习任务。
6. 用户开始学习并查看学习结果。

### 6.2 日常学习流程

1. 用户进入首页查看今日任务。
2. 用户完成新词学习。
3. 用户进入语法模块，查看句子拆解、语法点说明或完成句子练习。
4. 用户完成当天复习任务。
5. 用户参加测验巩固记忆。
6. 系统更新学习进度、掌握程度和打卡状态。

### 6.3 复习闭环流程

1. 用户学习新词后产生学习记录。
2. 系统根据掌握情况生成待复习任务。
3. 用户在复习中完成拼写、选择、释义匹配等题型。
4. 错误单词进入错词本，熟练单词降低复习频率。

## 7. 小程序端功能需求

## 7.1 页面结构

建议首版小程序包含以下页面：

- 启动页/登录页
- 首页
- 学习页
- 复习页
- 测试页
- 语法学习页
- 单词详情页
- 词书选择页
- 学习计划设置页
- 错词本页
- 收藏夹页
- 学习统计页
- 个人中心页

## 7.2 功能模块说明

### 7.2.1 登录与用户体系

功能要求：

- 支持微信一键登录。
- 首次登录自动创建用户档案。
- 支持获取并保存用户昵称、头像等基础信息。
- 支持登录状态续期。

业务规则：

- 用户唯一标识以微信 openid/unionid 或平台统一用户 ID 为准。
- 未完成登录的用户只能浏览部分公开内容，不可保存学习进度。

### 7.2.2 首页

功能要求：

- 展示今日学习任务总览。
- 展示今日新学单词数、待复习单词数、已完成进度。
- 展示连续打卡天数。
- 提供“继续学习”“开始复习”“进入测试”快捷入口。
- 展示当前词书和每日学习目标。

### 7.2.3 词书选择

功能要求：

- 支持查看词书列表。
- 支持按考试类型、学习阶段、难度进行筛选。
- 支持按词书名称关键词搜索。
- 支持查看词书简介、单词量、适用人群。
- 支持切换当前学习词书。

建议词书分类：

- 四级词汇
- 六级词汇
- 考研词汇
- 雅思词汇
- 托福词汇
- 小学/初中/高中词汇

### 7.2.4 学习计划

功能要求：

- 用户可设置每日学习数量，例如 10、20、30、50。
- 系统根据词书总量和学习目标估算完成周期。
- 支持暂停计划、重置计划、切换计划。
- 支持记录计划开始日期和完成进度。

业务规则：

- 同一时间默认仅允许一个生效中的主学习计划。
- 切换词书时，应提示是否保留旧进度或重新开始。

当前实现状态：

- 词书页已支持关键词搜索、考试/阶段筛选、难度层级筛选。
- 学习计划页已支持查看当前计划、更新每日目标、暂停/恢复计划。
- 切换词书时已支持选择保留旧进度或重新开始，计划每日目标会同步到用户设置。

### 7.2.5 单词学习

功能要求：

- 按学习计划展示今日待学单词列表。
- 每个单词展示拼写、音标、词性、中文释义、例句、发音。
- 支持标记“认识/不认识”“已掌握/未掌握”。
- 支持上一词、下一词切换。
- 支持将单词加入收藏夹。

交互要求：

- 学习卡片式展示，适合移动端操作。
- 支持自动播放发音或手动点击播放。
- 支持学习完成后的结果反馈。

### 7.2.6 复习功能

功能要求：

- 根据用户历史学习记录生成复习任务。
- 支持多种复习题型：
  - 释义选词
  - 单词选释义
  - 拼写填空
  - 听音辨词
- 单词、例句、听音辨词等所有朗读场景统一读取用户“学习设置”中的全局发音语速，首版提供 `1.0x`、`0.9x`、`0.8x`、`0.7x` 等选项。
- 复习结束后给出正确率和错题回顾。

业务规则：

- 首版可采用简化版间隔复习策略。
- 用户答错的单词优先进入近期复习队列。

### 7.2.7 测试功能

功能要求：

- 支持按词书范围发起小测验。
- 支持按最近学习内容生成测试。
- 支持固定题量，例如 10 题、20 题、50 题。
- 支持查看测试成绩和错题解析。

### 7.2.8 单词详情

功能要求：

- 展示单词详细信息。
- 展示发音、释义、例句、近义词、词形变化等扩展内容。
- 展示用户个人学习状态，如学习次数、最近复习时间、是否收藏、是否在错词本中。

### 7.2.9 错词本与收藏夹

功能要求：

- 错词本支持查看全部错词。
- 支持按最近错误时间、错误次数排序。
- 支持从错词本发起强化复习。
- 收藏夹支持手动添加和取消收藏。

### 7.2.10 学习统计

功能要求：

- 展示累计学习单词数。
- 展示累计复习次数。
- 展示测试次数和平均正确率。
- 展示最近 7 天/30 天学习趋势。
- 展示连续打卡天数和历史打卡记录。

### 7.2.11 个人中心

功能要求：

- 展示用户头像、昵称、当前词书、学习天数。
- 支持修改学习目标。
- 支持设置全局发音语速，影响单词、例句、复习听音、语法句子等可朗读内容。
- 支持查看隐私政策、用户协议、意见反馈；意见反馈由用户在小程序提交，管理员可通过后台网页查看与处理。
- 支持退出登录或清除本地缓存。

### 7.2.12 语法学习与长难句拆解

功能目标：

- 在底部导航栏新增“语法”主入口，作为独立于词书学习的专项模块。
- 通过句子级学习帮助用户理解主语、谓语、宾语、定语、状语、补语、从句、非谓语、时态、语态等语法成分与结构。
- 在查看词书之外，为用户提供一个可直接查句子结构、读整句释义、看拆解说明的语法学习场景。

功能要求：

- 语法主入口先展示功能分块首页，至少包含“自动拆句”“例句学语法”“语法总览”三个二级入口，避免把所有能力直接堆叠在同一页面。
- “自动拆句”用于用户输入任意英文句子后即时分析，展示原句、中文释义、句子主干、颜色标注和逐块拆解结果。
- “例句学语法”用于按专题、难度和关键词浏览句库例句，并在例句详情中完成主干阅读、结构走读和练习。
- “语法总览”用于按学习顺序提供基础语法地图，覆盖句子成分、基本句型、时态、语态、情态、非谓语、从句、比较结构等核心内容。
- 支持按语法主题浏览句子，例如时态、语态、主谓一致、名词性从句、定语从句、状语从句、非谓语、虚拟语气、比较结构、长难句拆解等。
- 支持按难度浏览句子，至少区分基础、进阶、考试/长难句三个层级。
- 支持按句子关键词、语法标签进行搜索。
- 每个句子需展示英文原句、中文翻译、核心语法点、难度、标签、可选音频。
- 支持对句子中的主语、谓语、宾语、表语、定语、状语、从句、连接词、时态/语态、非谓语结构等使用固定颜色体系进行标注。
- 支持点击句子中的单词或片段后，显示该片段的语法角色、中文说明、在整句中的作用和必要示例。
- 支持“整句浏览 → 分块拆解 → 语法点讲解 → 巩固练习”的渐进式学习流程。
- 支持长难句按意群分块展示，优先突出主干句，再补充修饰成分与从句层级。
- 支持句子练习，首版建议包括成分识别、时态判断、语序重组、选择题、译意匹配等题型。
- 支持记录用户的查看、练习、收藏和掌握情况。

交互要求：

- 语法首页需采用“先选功能，再进入学习”的信息架构，减少首次进入时的认知负担。
- 页面顶部提供颜色图例，明确不同颜色所代表的语法角色。
- 句子卡片需要支持点击高亮、局部解释弹层或底部讲解卡片。
- 需要提供“主干视图”和“完整视图”切换，降低用户首次阅读长难句的难度。
- 需要提供“上一句 / 下一句”连续学习模式，适合碎片化学习。
- 需要提供“我已理解 / 还不清楚”之类的轻量反馈，用于后续推荐与复习。

内容建设要求：

- 语法句库最终目标约 1000 句，覆盖校园、考试、日常交流、职场、阅读理解等常见场景。
- 每个句子必须提供真实、自然的中文释义，不允许使用占位释义。
- 每个句子必须提供完整的语法拆解说明，不允许仅展示标签而无解释。
- 长难句内容应包含主干提取、从句边界、修饰关系说明。
- 同一语法点下应同时包含基础句、常用句、易错句和考试类句子。

补充建设计划：

- 第一阶段：上线 200 句高频基础句，覆盖主谓宾、五大基本句型、一般现在时/过去时/将来时、常见状语和定语。
- 第二阶段：扩展到 500 句，补齐从句、被动语态、比较级、非谓语、主谓一致、强调句等重点语法。
- 第三阶段：扩展到 1000 句，加入考研、四六级、雅思、托福阅读中的长难句拆解与专项练习。

交互参考建议：

- 参考 Duolingo、Busuu 的“语法点说明 + 小步练习”模式，将规则说明与即时练习组合呈现。
- 参考 FluentU、LingQ 的“句子/字幕点击查看释义与整句理解”模式，将句子点击、整句翻译和上下文理解结合。
- 本项目建议采用“颜色标注 + 点击讲解 + 主干切换 + 小练习”四段式交互，避免只做纯文本语法课。

AI 一期现状说明：

- 自动拆句页已接入第一期 AI 老师能力。
- 当前已支持“规则拆句 + AI 讲解 + 关联语法点 + 相似例句 + AI 微练习 + 语法追问”。
- 当前已补齐 AI 讲词老师、首页 AI 学习教练、AI 错词复盘，以及面向 Agent/MCP 的内部工具调用接口。
- 跟读识别和发音评分暂不纳入当前版本，待微信后台插件授权完成后再作为最后阶段实现。

### 7.2.13 AI 助学与智能教练规划

功能目标：

- 将小程序从“内容展示型学习工具”升级为“内容 + 智能讲解 + 个性化陪练 + 学习教练”的复合学习工具。
- 让 AI 与词书、语法、错词本、学习计划、统计数据联动，而不是作为孤立聊天窗口存在。

总体原则：

- AI 必须优先基于项目内已有词书、语法句库、学习记录和错词数据回答，降低幻觉风险。
- AI 功能默认通过后端统一输出，小程序端只负责展示，不直接调用第三方模型。
- AI 输出应优先服务学习闭环，例如“解释 -> 追问 -> 练习 -> 记录 -> 推荐下一步”。
- 单词学习页和单词详情页默认展示本地讲词；AI 讲词由用户主动开启，不应把本地讲词结果伪装成 AI 结果。
- 首页 AI 学习教练默认关闭；用户主动开启后，后端基于当前计划、已学词、错词、复习压力和统计趋势生成个性化建议。

当前已实现 AI 功能清单：

- AI 讲词老师：在单词详情页、学习页中解释词义、易混词、记忆法、搭配和推荐原因。
- AI 错词复盘：对错词本做错误归因、薄弱点总结、强化复习建议和专项微练习。
- AI 学习教练：在首页与学习计划中输出今日建议、学习强度、优先级、学习顺序和阶段性总结。
- AI 周报/月报：对学习趋势、掌握变化、错题变化和语法薄弱项做自然语言总结。
- AI 语法总览导学：根据用户水平与历史掌握情况，动态推荐适合的语法章节与例句专题。
- AI 连续追问会话：支持围绕同一句子或同一语法点进行多轮问答，而非只支持单轮问答。
- AI 写作批改：支持句子和短文纠错、语法解释、自然表达替换和对应语法点回链。
- AI 写作题目与范文：按难度、主题和文体生成写作题、提纲、参考范文、评分维度和可用表达。
- AI 中译英/英译中训练：支持翻译练习、答案评分、替代表达和即时纠错。
- 结构化 RAG 检索：在词库、语法点和句库中检索相关上下文，并生成带来源的学习回答。
- 轻量向量 RAG：基于项目词库、语法点和句库构造本地 token 向量评分，不依赖外部向量库即可做相似资料召回。
- AI 情景文本对话：支持日常、餐厅、课堂、面试、旅行、购物、酒店、机场等场景的文本对话任务卡、角色卡、表达纠错和下一句建议。
- AI 多 Agent 协作：支持语法老师、单词老师、学习教练、写作批改等多智能体分工。
- AI 会话记录与反馈：保存 AI 会话、提示词版本、运行日志和用户反馈，管理员可在后台查看。
- AI 评测与观测：支持 AI 消息/反馈质量摘要、运行日志、响应缓存、缓存命中统计、基础限流和 RAG 召回覆盖评测。
- AI 报告历史增强：支持周报/月报历史查看，以及和上一期的简要对比摘要。
- MCP 工具层：已提供内部 HTTP 工具调用入口和 manifest/resources/prompts 描述接口，支持读取学习计划、今日任务、到期复习、错词本、词汇详情、学习快照、语法点、句子详情、结构化 RAG、轻量向量 RAG、写作题生成、语法导学、学习报告、多 Agent 简报和 AI 观测摘要。

待实现 AI 功能清单：

- 生产级向量 RAG：在轻量本地向量评分基础上补充 embedding 模型、向量索引、知识切片管理和更稳定的召回评测。
- 多 Agent 工作流编排：将当前多老师简报升级为可执行的 LangGraph 状态机，例如诊断 -> 练习 -> 复盘 -> 调整计划。
- 深度报告增强：继续补趋势归因、可导出的学习档案和更细的周期对照。
- AI 口语老师：围绕发音、语调、连读弱读和表达自然度提供反馈；该能力依赖微信语音识别/同类插件授权，当前版本后置。

AI 功能优先级建议：

- 已落地：AI 讲词老师、AI 学习教练、AI 错词复盘、AI 周报/月报、AI 报告历史对比、AI 语法导学、AI 写作批改、AI 写作题目与范文、翻译训练、AI 情景文本对话任务模板、连续追问会话、AI 会话反馈、结构化 RAG、轻量向量 RAG、RAG 召回评测、MCP tools/resources/prompts 描述接口、多 Agent 简报、AI 运行日志、缓存、限流和基础质量摘要。

## 10. 业务规则

### 10.1 学习计划规则

- 用户默认只能有一个进行中的主计划。
- 每日学习目标必须大于 0。
- 若当日任务未完成，可累积到次日，但首页需明确区分“今日新任务”和“历史未完成任务”。

### 10.2 单词掌握规则

- 首版掌握度可采用 0 到 5 的等级模型。
- 用户答对次数越多，掌握度越高。
- 用户答错后，掌握度下降或保持不变。

### 10.3 错词规则

- 复习或测试答错的单词自动进入错词本。
- 同一单词重复答错时增加错误次数，不重复插入多条独立记录。

### 10.4 打卡规则

- 用户当日完成学习目标或达到预设阈值后，可视为打卡成功。
- 连续打卡中断后重新计算连续天数。

### 10.5 语法学习规则

- 用户查看语法句子时，应默认可见整句中文释义和核心语法说明，不以是否学过该句作为显示前提。
- 语法标注颜色需全站保持一致，同类语法角色不可在不同页面使用不同颜色语义。
- 同一句子可关联多个语法点，但必须配置一个主语法点作为列表分类依据。
- 长难句必须支持主干提取和分块拆解，不能只展示整句原文与翻译。
- 语法学习记录可参与统计，但不应干扰单词学习主计划的进度计算。

### 10.6 AI 助学规则

- AI 输出必须优先结合项目内知识库、语法句库、词书数据和用户学习状态，不应脱离业务数据自由发挥。
- AI 功能必须保留非 AI 的基础学习链路，不能让登录、学习、复习、语法浏览等核心路径依赖 AI 才能使用。
- AI 回答需标记当前来源模式，例如规则分析、AI 增强、知识库检索增强等，避免用户误判结果可靠性。
- AI 练习题、讲词建议、学习教练建议应可追溯到当前句子、单词、语法点或用户状态，不允许给出完全无上下文的泛化建议。
- AI 生成内容若写入长期数据，应经过显式确认或后台审核，不能直接覆盖基础词库与语法库。
- AI 请求需支持超时、重试、降级和缓存策略；非 AI 基础链路可继续使用本地/规则内容，但 AI 面板在请求中、超时或未配置时不得继续显示旧的本地讲解，避免用户混淆。

## 11. 非功能需求

### 11.1 性能要求

- 首页接口响应时间目标小于 1 秒。
- 常规列表接口响应时间目标小于 800 毫秒。
- 单次学习记录提交接口应支持高频调用。

### 11.2 安全要求

- 所有接口必须使用 HTTPS。
- 服务端必须校验用户身份，不得仅依赖前端传参。
- 重要用户数据需进行访问控制与日志记录。

### 11.3 可维护性要求

- API 采用统一版本管理。
- 业务日志、错误日志、访问日志需可追踪。
- 代码结构需支持后续扩展管理后台与会员体系。
- Django 项目需区分开发、测试、生产环境配置。
- Django app 拆分应与业务模块保持一致，避免所有逻辑集中在单一应用中。

### 11.4 技术与部署要求

- 后端服务必须可基于 Django 项目独立启动，并支持本地开发与生产部署。
- 后端服务必须能够稳定连接 MySQL 8 数据库并完成迁移、查询、写入操作。
- 数据库连接配置需支持环境变量管理，不得将生产库账号密码硬编码到代码中。
- 需要预留静态资源、媒体资源、日志目录和配置文件的标准化管理方式。

### 11.5 数据统计要求

- 关键行为需埋点，包括登录、选词书、开始学习、完成学习、开始复习、完成测试、打卡。
- 支持后续输出日活、留存、学习完成率等基础运营指标。

### 11.6 内容质量要求

- 语法句库中的英文句子、中文翻译、语法说明和标注结果需经过人工校对。
- 不允许出现“示例句占位文本”“临时翻译”“未标注完成”的线上内容。
- 语法句库扩充到 1000 句时，需建立批量校验机制，检查缺失翻译、缺失标注、颜色映射错误和句子重复问题。

## 12. 验收标准

### 12.1 小程序端验收标准

- 用户可完成微信登录并正常进入系统。
- 用户可选择词书并设置每日学习目标。
- 用户可看到今日学习任务并完成新词学习。
- 用户可从导航栏进入“语法”模块，查看句子中文释义、语法标注和拆解说明。
- 用户可完成至少一种语法句子练习，并看到自己的语法学习进度。
- 用户可完成复习和测试流程。
- 用户可查看错词本、收藏夹和学习统计。

### 12.2 后端 API 验收标准

- 所有核心接口可正常联调。
- 用户学习进度、复习记录、测试结果可正确保存。
- 语法主题、句子详情、句子标注和语法学习记录接口可正常返回与保存。
- Token 鉴权生效，未登录用户无法访问私有数据。
- 接口返回结构统一，错误码明确。
- Django 项目可正常完成数据库迁移并成功连接 MySQL 8。
- 核心业务表可完成建表、索引创建和基础数据读写验证。

## 14. 风险与待确认事项

- 词书和单词数据来源是否已有现成资源。
- 发音音频是第三方资源还是自建资源。
- 首版是否需要管理员在后台维护词书与单词。
- 语法句库 1000 句的数据来源、授权方式和人工标注成本需要提前确认。
- 语法标注采用人工标注、规则辅助标注还是半自动校对流程，需在开发前确定。
- 微信登录使用 openid 还是 unionid 作为长期主键策略。
- 复习算法首版采用简化规则还是引入更完整的记忆曲线算法。
- Django 版本、部署方式和运维环境仍需在开发启动前统一确定。
- AI 助学阶段使用的模型、向量库、embedding 模型、观测平台和成本控制策略需在开发前统一。
- AI 回答的质量评测标准、用户反馈闭环和数据安全边界需在大规模上线前确认。
